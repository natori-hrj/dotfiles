## 適用タイミング

コードを書いた後、ユーザーがコミットする前に必ずこのチェックリストを確認する。

## チェックリスト

### 1. docstring/コメントの正確性
- [ ] docstringが実際のロジックと一致しているか
- [ ] 条件分岐がある場合、すべてのケースがdocstringに反映されているか
- [ ] 「required」「必須」などの表現が正確か（どの環境で必須か明記）
- [ ] 例外的なケースや前提条件が記載されているか

### 2. ログ/メッセージの正確性
- [ ] 全分岐で適切なメッセージが出力されるか
- [ ] else/except ブロックのメッセージが全ケースに当てはまるか
- [ ] 複数の条件で到達する分岐は、すべてのケースを考慮したメッセージか
- [ ] エラーメッセージが原因特定に十分な情報を含むか

### 3. パス/ディレクトリ処理
- [ ] subprocess実行時にcwdが明示されているか
- [ ] 相対パスを使う場合、基準ディレクトリが明確か
- [ ] ファイルパスは環境に依存しない方法で構築されているか
- [ ] `__file__`からのパス解決が必要な場合、正しく実装されているか

### 4. 取得したデータの活用
- [ ] 取得した変数/フラグが実際にロジックで使われているか
- [ ] 使われていない変数がある場合、削除するか使用するか判断
- [ ] 状態チェックで取得した情報が判定に反映されているか

### 5. エッジケース/分岐網羅
- [ ] if/elif/else の全分岐で適切な処理がされているか
- [ ] 空リスト、None、0、空文字列などの境界値を考慮しているか
- [ ] 複合条件（A and B and C）で、各組み合わせを検討したか
- [ ] 異常系（複数行、不正値など）の検知と警告があるか

### 6. セキュリティ
- [ ] ユーザー入力をそのまま信頼していないか
- [ ] クライアントから受け取るべきでないフィールドを受け取っていないか
- [ ] 環境変数のデフォルト値が安全か
- [ ] 本番環境での破壊的操作に適切なガードがあるか

### 7. 堅牢性
- [ ] 外部コマンド実行時にvenv/パスの問題を考慮しているか
- [ ] DB接続などのリソースが適切にクローズされるか
- [ ] リトライや例外処理が適切か
- [ ] タイムアウトの考慮が必要な箇所にあるか

### 8. React/フロントエンド固有
- [ ] setState後に即座にその値を参照していないか（非同期更新のため古い値になる）
- [ ] useCallbackの依存配列が正しいか
- [ ] 未使用のパラメータや変数がないか（将来用でも削除推奨）
- [ ] 新しいhookにテストがあるか

### 9. テストの品質
- [ ] モックデータが実際の実装と一致しているか（モデル名、定数値など）
- [ ] 新しいメソッド/エンドポイントにテストがあるか
- [ ] 成功ケースと失敗ケースの両方をテストしているか

## 実行方法

コード変更後、コミット前に以下を実行：

1. 上記チェックリストを変更箇所に適用
2. 問題があれば修正
3. 修正内容をユーザーに報告
4. ユーザーがコミット

## 新しい指摘からの学習

Copilotから新しいパターンの指摘を受けた場合：

1. 指摘内容を分析し、既存のチェック項目でカバーできるか確認
2. カバーできない場合、新しいチェック項目をこのファイルに追加
3. 学習元セクションにPR番号と指摘内容を記録

**重要**: ユーザーからCopilotの指摘を共有された際、このファイルを自動で更新すること。
